# Phase 1.1 问题清单

## 问题 1：Phase1_Report 宣称新增目录但仓库中不存在
- **证据**：`src/.../docs/Phase1_Report.md:20-38` 声称一次性创建包含 `runtime/、calculator/、tuning/、client/、events/、ui/、storage/` 的新目录结构。
- **实际情况**：`find src/main/java/net/tigereye/chestcavity/compat/guzhenren/item/hun_dao -maxdepth 1 -type d` 仅列出 `behavior、combat、docs、fx、middleware、runtime、storage、tuning`；没有 `calculator/、client/、events/、ui/` 目录。
- **影响**：计划/报告无法真实反映当前结构，后续 Phase 参考这些目录会失败，阻塞架构对齐验证。
- **建议**：按计划创建占位目录（可放 README）或修正文档并更新计划状态。

## 问题 2：“所有资源操作通过接口”与实际实现不符
- **文档位置**：`Phase1_Report.md:40-173` 宣称“行为类不再直接依赖 GuzhenrenResourceBridge 或 ResourceOps，所有资源操作经 HunDaoResourceOps”。
- **代码现状**：除 `HunDaoSoulBeastBehavior.java` 外，其余魂道行为仍直接调用底层实现。例如：
  - `behavior/XiaoHunGuBehavior.java:84-105` 直接 `GuzhenrenResourceBridge.open()` 并调用 `ResourceOps.tryAdjustDouble()`。
  - `behavior/DaHunGuBehavior.java:72-99` 同样直接操纵 `ResourceHandle`。
  - `behavior/GuiQiGuBehavior.java`、`behavior/TiPoGuOrganBehavior.java` 等文件也存在同样依赖（可通过 `rg -n "GuzhenrenResourceBridge" .../item/hun_dao/behavior` 验证）。
- **影响**：Phase 1 目标（接口解耦）未完成，模块仍难以替换资源层实现，文档与事实不符，后续 Phase 设计依据失真。
- **建议**：
  1. 真正为这些行为引入 `HunDaoResourceOps`／`HunDaoOpsAdapter` 注入，并移除直接依赖。
  2. 更新 Phase1_Report，明确当前仅有 `HunDaoSoulBeastBehavior` 完成解耦，剩余部分需在 Phase 1.x/2 继续推进。
